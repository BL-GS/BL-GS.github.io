Paper: NoHype

发表于2023-10-04|更新于2023-10-06|Virtualization
|阅读量:

Note：这是 2010 年的文章，当时很多技术包括 IO 虚拟化和硬件虚拟化的支持还不完善，因此会有些 outdated。但是这篇文章也只是提出一个架构，甚至没有 evaluation。

Background

  • 存在众多客户和服务器，基础设施的管理需要高度自动化。在 multi-tenancy 场景下，多个客户共享同一台服务器的资源
  • multi-tenancy 导致了严重的安全隐患。通过攻击虚拟化层，恶意程序能够获得其他客户机的信息甚至控制权，也就是侧信道攻击
  • hypervisor 变得越来越复杂，非常难以保证其安全

Main Idea

为了安全，干脆完全移除虚拟化层（virtualization layer），同时保留虚拟化的特征

  • 移除虚拟化层能够避免恶意程序攻击虚拟化层，从而导致侧信道泄露
  • 移除虚拟化层能够消除一定开销，即虚拟化税（virtualization tax）

Assumption

  • 云服务商不是恶意的
  • 云管理软件是安全的

Design

  • 每个处理器核心专注服务于一个虚拟机。也就是，同一个处理器上不会因为虚拟机切换做 context switch
  • 客户机在运行时不与管理软件交互，也不会被其中断
  • 每个 VM 只能访问被分配的内存，通过多核内存控制器（MMC）和硬件页表机制来限制内存映射
  • 通过硬件公平地划分物理内存
  • 虚拟 IO 设备，公平地分配带宽和资源
  • 通过流控制机制来实现 IO 总线的速率控制

Processor

  1. Starting a VM
    1. 系统管理器从云管理器接收到命令，包括 VM 的描述和磁盘镜像的位置
    2. 分配并初始化 VM 内存和磁盘空间
    3. 将 VM 内存和磁盘空间从系统管理器卸载
    4. 向 VM 处理器发送 start IPI，在虚拟化特权级下启动 core manager，初始化资源。最后通过 VM exit 启动客户操作系统
  2. Stopping a VM
    1. 系统管理器从云管理器接收到停止命令
    2. 向 VM 处理器发送 stop IPI，从而切换到虚拟化特权级，执行 core manager
    3. core manager 处理资源
    4. 让处理器闲置
  3. Aborting a VM
    1. 在接收到非法操作时，可以使 VM 崩溃
    2. 切换到虚拟化特权级，执行 core manager
    3. 向系统管理器发送 end IPI，处理资源
    4. 让处理器闲置
  4. Live Migration a VM
    1. 系统管理器从云管理器接收到迁移命令
    2. 内存管理器记录脏页
    3. 向 VM 处理器发送 migrate IPI
    4. 通过迭代快照的方式，逐步迁移 VM 数据，直到差距较小，直接迁移 VM

Hardware Utilization

  • 通过 APIC，处理器核心之间传递 IPI。通过终端掩码，避免接收到来自其他处理器核心的中断
  • 通过 Extended Page Tables(EPT) 机制隔离内存空间
  • 通过脏页追踪，迭代复制实现热迁移（年代限制）
  • IOMMU 限制 IO 设备的访问，MSI 和 MSI-X 扩展处理设备中断
  • SR-IOV 实现设备虚拟化

Security

  • 虚拟机上下文切换：通过核心不切换 VM 解决
  • 中断正在运行 VM 的处理器：通过硬件掩码实现屏蔽 IPI 解决
  • IO 不公平占用：通过多核内存控制器的配置解决
  • 侧信道攻击：由于处理器隔离，消除了 L1 Cache 的共享问题；此外内存和 IO 隔离也能够一定程度上限制侧信道攻击
  • 只有系统管理器能够配置设备虚拟化

Problem

  • 不能够更细粒度地划分资源
  • 不能超订阅（over-subscribe）物理资源。也就是，卖出的资源不能多于已有的资源，毕竟不能通过虚拟化层把资源“模拟”出来。
  • 硬件可能会有 bug
  • 只隔离了 L1 Cache，没有详细描述共享 Cache 的保护机制
  • Interrupt Command Register(ICR) 的配置并没有说明独占性，客户操作系统可能通过配置该寄存器，脱离系统管理器的控制
  • 比如利用了 HPET 支持多个时钟，但是时钟数量是有限的，文中并没有阐述其上限问题
