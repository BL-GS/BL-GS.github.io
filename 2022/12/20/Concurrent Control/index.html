Concurrent Control

Posted on 2022-12-20 In Transaction
Introduction of some concurrent control.

并发控制

我们可以通过多种 并发控制策略 来保证：即使有多个事务并发执行时，不管操作系统在这些事务之间如何分配分时资源，都只产生可接受的调度。最常用的机制有：两阶段锁和快照隔离

 

两阶段锁 two-phase locking

这里只考虑两种模式：共享模式和互斥模式（参考读写锁）

要求每个事务都要根据自己将对数据项 \(Q\) 执行的操作类型 申请(request) 其上的锁。事务只有再并发控制管理器 授予(grant) 它所需要的锁后才能继续操作。

可能会发生 死锁(deadlock) 的情况，当这种情况发生时，系统必须回滚两个事务中的一个

避免事务的饿死：

当一个事务 \(T_i\) 申请对一个数据源 \(Q\) 加特定模式 \(M\) 的锁时，并发控制管理器授权该锁的条件是：

  • 在 \(Q\) 上持有与 \(M\) 冲突模式的锁的其他事务是不存在的
  • 正在等待对 \(Q\) 加锁且先于 \(T_i\) 提出其锁申请的事务是不存在的

 

两阶段锁协议

保证可串行化的一种协议是 两阶段封锁协议 two-phase locking protocol ，该协议要求每个事务分为两个阶段提出加锁和解锁申请

  • 增长阶段(growing phase) ：一个事务可以获得锁，但不能释放任何锁
  • 缩减阶段(shrinking phase) ：一个事务可以释放锁，当不能获得任何新锁

开始，一个事务处于增长阶段，事务根据需要获取锁。一旦该事务释放了一个锁，它就进入了缩减阶段，并且不能再发出加锁请求。

在调度中该事务获得其最后锁的位置（事务增长阶段的结束点）称为该事务的 封锁点(lock point)。这样，事务可以根据它们的封锁点进行排序，这种次序就是这些事务的一种串行化次序。两阶段锁不保证不产生死锁

级联回滚可以通过将两阶段锁修改成为 严格两阶段锁协议 strict two-phase locking protocol 来避免。这种协议要求保证未提交事务所写的任何数据在该事务提交之前均以排他模式封锁，以防止其他事务读取这些数据。

强两阶段封锁协议 rigorous two-phase locking protocol 要求在事务提交之前保留所有的锁。

 

封锁的实现

这里提出一种将共享锁升级为排他锁、将排他锁降级为共享锁的机制——锁转换(lock conversion)

  • 升级(upgrade)：从共享模式到排他模式
  • 降级(downgrade)：从排他模式到共享模式

从某种程度上，升级只能发生在增长阶段，降级只能发生在缩减阶段

锁管理器(lock manager) 可以实现为一个进程，它从事务处接受消息并以应答形式发送消息。它为目前已加锁的每个数据项维护一个记录的链表，每一个请求对应链表中的一条记录，并按请求到达的顺序排序。对于一个数据项，链表中的每一条记录记载了由哪个事务提出的请求，以及请求的锁类型，是否被授予等。

  • 当一条锁请求消息到达时，如果相应数据项的链表存在，则它在该链表的末尾增加一条记录；否则，创建一个仅包含对于该请求的记录的新链表。

    在没有被封锁的数据项上的锁请求总是被授予的，但是如果请求当前已被封锁的一个数据项上的锁，那么只有该请求与当前持有的锁相容并且先前的请求都已经被授予锁的情况下，才为其授予锁，否则等待。

  • 当锁管理器收到来自一个事务的解锁消息时，将对应于该事务的数据项链表中的记录删除，然后检查随后的记录，若有则判断请求是否能够被授权。

  • 如果一个事务中止，则锁管理器删除该事务产生的任何等待请求。

基于图的协议 通过根据数据项的偏序构建有向无环图，成为 数据库图(database graph)。这里仅讨论一种称为 树型协议(tree protocol) 的简单协议，该协议被限制为只使用排他锁。

  • \(T_i\) 的首次封锁可以在任意数据项上进行
  • 此后，\(T_i\) 可以对一个数据项 \(Q\) 加锁的前提是 \(T_i\) 当前持有 \(Q\) 的父项上的锁
  • 对数据项解锁可以随时进行
  • 一个数据项被 \(T_i\) 加锁并解锁后，\(T_i\) 不能再对其上锁。
  • 为了保证可恢复性和无级联性，还可以修改为事务结束之前不能释放排他锁。

提高并发度的一种方法是：

对于每一个未提交写的数据项，我们记录是哪一个事务最后对该数据项执行过写操作的。每当一个事务 \(T_i\) 执行对未执行数据项的读操作时，我们就在最后对该数据项执行写操作的事务上记录一个 \(T_i\) 的 提交依赖(commit dependency)。在事务 \(T_i\) 的提交依赖的所有事务提交完成之前，\(T_i\) 不允许提交。其中任一个中止，那么\(T_i\)也需要中止。

这种协议保证了冲突可串行化，还保证了不会产生死锁，也就无需回滚。

缺点是一个事务可能必须给它根本不访问的数据项加锁，降低了并发度。同时可能存在不能通过树型协议来得到的冲突可串行化调度。

 

死锁处理

两种处理死锁的主要方法：死锁预防 deadlock prevention 和 死锁检测与死锁恢复 deadlock detection & deadlock recovery

这两种方法都有可能引起事务回滚，若系统进入死锁的概率较高，通常使用预防机制，否则使用检测和恢复机制会更加有效。

 

死锁预防

预防死锁有两种方法：

  • 通过对封锁请求进行排序，或要求同时获得所有的锁来保证不会循环等待
  • 每当等待可能导致死锁时，执行事务回滚而不是等待。

最简单的方式要求事务执行前封锁它所有数据项，但是事务开始之前很难知道它会封锁哪些数据项；同时数据项的使用率可能很低，即饥饿

另一种方式是对数据项施加一种次序，同时要求事务只能按照次序规定的顺序来封锁数据项。

第二种方式是使用抢占和事务回滚。在抢占机制种，若一个事务 \(T_j\) 所申请的锁被事务 \(T_i\) 持有，则授予 \(T_i\) 的锁可能通过回滚 \(T_i\) 而被 抢占(preempted) ，并将该锁授予 \(T_j\)。为了控制抢占，可以通过一个计数器或者基于系统时钟来给每个事务赋予一个唯一的时间戳。若一个事务被回滚，则重启时保留原有时间戳。：

等待-死亡机制 wait-die 是一种非抢占技术。当事务 \(T_i\) 申请的数据项当前被 \(T_j\) 持有时，仅当 \(T_i\) 的时间戳小于 \(T_j\) 的时间戳的情况下，才允许 \(T_i\) 等待，否则 \(T_i\) 死亡。

伤害-等待机制 wound-wait 是一种抢占技术。当事务 \(T_i\) 申请的数据项当前被 \(T_j\) 持有时，仅当 \(T_i\) 的时间戳大于 \(T_j\) 的时间戳的情况下，才允许 \(T_i\) 等待，否则 \(T_j\) 回滚。

锁超时 lock timeout。申请锁至多等待一段指定的时间。若在那段时间内锁尚未授予该事务，则称该事务超时，并且该事务执行回滚并重启。

 

死锁检测与恢复

需要定期调用检查系统状态的算法以确定是否发生了死锁。如果发生了死锁，系统必须尝试从死锁中恢复。必须有：

  • 维护当前将数据项分配给事务的有关信息，以及任何悬而未决的数据项请求信息
  • 提供一个使用这些信息来判断系统是否进入死锁状态的算法。
  • 当检测算法判定存在死锁时，则从死锁中恢复。

死锁可以用一种称为 等待图(wait-for graph) 的有向图来描述。当且仅当等待图中包含回路时，系统中存在死锁。调用检测算法的频率以及受死锁影响的事务数量决定我们该何时调用检测算法。

当检测算法判定存在死锁时，系统必须从死锁中恢复(recover)。解决死锁最常用的解决方案是回滚一个或者多个事务。

  • 选择牺牲者。给定一组死锁的事务，必须决定回滚哪一个事务来打破死锁。
  • 回滚。一旦决定了必须回滚哪一个事务，还需要决定这个事务应该回滚多远。
  • 饿死。可能发生同一个事务一直被选为牺牲者的情况。

 

基于时间戳的协议

 

时间戳

对于系统中的每个事务 \(T_i\)，我们把一个唯一的、固定的时间戳和它关联起来，记为 TS(\(T_i\))。该时间戳是在事务 \(T_i\) 开始执行之前由数据库系统所赋予的。可以采用两种方式实现：

  • 使用 系统时钟(system clock) 的值作为时间戳。也就是说，事务的时间戳等于其进入系统时时钟的值
  • 使用 逻辑计数器(logical counter)，每赋予一个新的时间戳，该计数器就递增计数。也就是说，事务的时间戳等于其进入系统时计数器的值。

将每个数据项和两个时间戳相关联：

  • W-timestamp(Q) 表示成功执行 write(Q) 的任意事务的最大时间戳
  • R-timestamp(Q) 表示成功执行 read(Q) 的任意事务的最大时间戳

 

时间戳排序协议 Timestamp-ordering protocol

  • 假设事务 \(T_i\) 发出 read(Q)
    • 若 TS(\(T_i\)) < W-timestamp(Q)，则 \(T_i\) 需要读取的 Q 值已被覆盖。因此，拒绝该 read 操作，执行回滚。
    • 若 TS(\(T_i\)) >= W-timestamp(Q)，则执行 read 操作，并且 R-timestamp(Q) 被设置为 R-timestamp(Q) 和 TS(\(T_i\)) 的最大值。
  • 假设事务 \(T_i\) 发出 write(Q)
    • 若 TS(\(T_i\)) < R-timestamp(Q)，则 \(T_i\) 需要读取的 Q 值是先前所需要，并且系统已经假定该值绝不会再产生。因此，拒绝该 write 操作，执行回滚。
    • 若 TS(\(T_i\)) < W-timestamp(Q)，则 \(T_i\) 试图写入的 Q 值是过时的，因此系统拒绝这个 write 操作并回滚 \(T_i\)

当事务 \(T_i\) 由于发出 read 或 write 操作而被并发控制机制回滚，那么系统会赋予它一个新的时间戳并重新启动它。

时间戳排序协议保证了冲突可串行化，这因为冲突的操作是按照时间戳顺序进行处理的。同时保证没有死锁，因为没有事务会发生等待。但是可能会导致长事务饿死的现象。

该协议还可能产生不可恢复的调度，需要通过以下几种方式之一来保证：

  • 通过在事务末尾一起执行所有的写操作能够保证可恢复性和无级联性。

    这些写操作必须具有下述意义的原子性：在写操作正在执行的过程中，任何事物都不允许访问已经写过的任何数据项。

  • 可恢复性和无级联性也可以通过使用一种受限的封锁形式来保证。

    由此，对未提交的数据项的读操作就被推迟到更新该数据项的事物提交之后

  • 可恢复性可以通过跟踪未提交写操作来单独保证。

    如果事物 \(T_i\) 读取了其他事务所写的值，那么只有在任何写事务都提交之后，\(T_i\) 才允许提交。

如果时间戳排序协议只用于元组，那么该协议将容易受到幻象现象的影响。为避免这个问题，时间戳排序协议可以应用于事务所读取的所有数据，包括关系元数据和索引数据

 

Thomas 写规则

对时间戳协议做一种修改。假设事务 \(T_i\) 发出 write(Q)

  • 若 TS(\(T_i\)) < R-timestamp(Q)，则 \(T_i\) 产生的 Q 值是先前所需要的，并且已经假定该值永远不会被产生了。因此，系统拒绝该 write 操作并回滚 \(T_i\)
  • 若 TS(\(T_i\)) < W-timestamp(Q)，则 \(T_i\) 试图写入的 Q 值已过时。因此，这个 write 操作可被忽略。
  • 其他情况，系统执行 write 操作，并将 W-timestamp(Q) 设置为 \(TS(T_i)\)

区别在于第二条，不再要求回滚，而是直接忽略过时的写操作。

 

基于有效性检查的协议 Validation protocol

在大部分事务是只读事务的情况下，事务之间发生冲突的概率很低。因此即使在没有并发控制机制监控的情况下，许多这样的事务的执行也不会破坏系统的一致性状态。

有效性检查协议 要求每个事务 \(T_i\) 在其生命周期中按两个或者三个不同的阶段执行，这取决于该事务是一个只读事务还是一个更新事务。

  • 读阶段。在这一阶段中，系统执行事务 \(T_i\)。它读取各数据项的值并将它们保存在 \(T_i\) 的局部变量中。\(T_i\) 的所有写操作都是对局部的临时变量进行的，并不对数据库进行真正的更新。
  • 有效性检查阶段。对事物 \(T_i\) 进行有效性检查的测试。决定是否允许 \(T_i\) 继续执行到写阶段而不违反可串行性。如果事物的有效性检查的测试失败，则中止这个事务。
  • 写阶段。若事务 \(T_i\) 通过了有效性检查的测试，则保存 \(T_i\) 所执行的任何 write 操作结果的临时局部变量间拷入数据库。

将三个不同的时间戳与每个事务 \(T_i\) 关联。

  1. StartTS(\(T_i\))：事务 \(T_i\) 开始其执行的时间
  2. ValidationTS(\(T_i\))：事务 \(T_i\) 完成其读阶段并开始其有效性检查阶段的时间
  3. FinishTS(\(T_i\))：事务 \(T_i\) 完成其写阶段的时间

我们利用 ValidationTS(\(T_i\)) 时间戳的值，通过时间戳排序技术来决定可串行化的次序。事务的有效性检查要求满足 TS(\(T_k\)) < TS(\(T_i\)) 的所有事务 \(T_k\) 必须满足以下两个条件之一：

  • FinishTS(\(T_k\)) < StartTS(\(T_i\))。因此 \(T_k\) 在 \(T_i\) 开始之前就完成其执行，故可串行化次序实际上是保证的
  • \(T_k\) 所写的数据项集合与 \(T_i\) 所读的数据项集合并不相交，并且在 \(T_i\) 开始其有效性检查阶段强，\(T_k\) 就完成了其写阶段(StartTS(\(T_i\)) < FinishTS(\(T_k\)) < ValidationTS(\(T_i\)))。这个条件保证 \(T_k\) 和 \(T_i\) 的写并不重叠。因为 \(T_k\) 的写并不影响 \(T_i\) 的读，又因为 \(T_i\) 不可能影响 \(T_k\) 的读，从而可串行化次序是可以保证的。

有效性检查机制自动预防了级联回滚，因为只有发出写操作的事务提交之后实际的写才发生。然而，存在长事务被饿死的可能性。

有效性检查机制中，由于事务是乐观执行的，并假定它们能够完成执行并且最终是有效的，因此也称为乐观的并发控制 Optimistic concurrency-control 机制。与之相反，封锁和时间戳排序是悲观的。

 

多版本机制

在 多版本并发控制 multiversion concurrency control 机制中，每个 write(Q) 操作创建 Q 的一个新版本。当事务发出一个 read(Q) 操作时，并发控制管理器选择 Q 的一个版本进行读取。

 

多版本时间戳排序

对于系统中每个事务 \(T_i\)，我们将一个静态的唯一性时间戳与之关联，记为 TS(\(T_i\))，数据库系统在事务开始执行之前赋予这个时间戳。

对于每个数据项 Q，有一个版本序列 <\(Q_1, Q_2, \cdots, Q_m\)> 与之关联，每个版本 \(Q_k\) 包含三个数据字段：

  1. Content：为 \(Q_k\) 版本的值
  2. W-timestamp(Q)：创建 \(Q_k\) 版本的事务的时间戳
  3. R-timestamp(Q)：是所有成功地读取过 \(Q_k\) 版本地任意事务的最大时间戳。

事务通过发出的 write(Q) 操作来创建数据项 Q 的一个新版本 \(Q_k\)。该版本的内容字段保存 \(T_i\) 所写的值。系统将 W-timestamp 和 R-timestamp 初始化为 TS(\(T_i\))。每当一个事务 \(T_j\) 读取 \(Q_k\) 的内容且 R-timestamp(\(Q_k\)) < TS(\(T_j\)) 时，系统就更新 \(Q_k\) 的 R-timestamp 的值。

多版本时间戳排序机制 (multiversion timestamp-ordering scheme)

其中 \(Q_k\) 代表拥有小于或等于 TS(\(T_i\)) 的最大时间戳的版本。

  • 如果事务 \(T_i\) 发出 read(Q)，则返回的值是 \(Q_k\) 版本的内容
  • 如果事务 \(T_i\) 发出 write(Q)，且若 TS(\(T_i\)) < R-timestamp(\(Q_k\))，则系统回滚事务 \(T_i\)。另一个方面，若 \(TS(T_i) = W-timestamp(Q_k)\) ，则系统覆盖 \(Q_k\) 的内容；否则，创建 Q 的一个新版本。

对于数据项，其 W-timestamp 为 t 的 \(Q_i\) 版本的有效区间：

  • 若其为最新版本，则为 \([t, \inf]\)
  • 否则，Q 的下一个版本的时间戳为 \(s\)，为\([t, s)\)

当所有事务的时间戳都大于 \(Q_k\) 版本的时间戳，则 \(Q_{k-1}\) 允许被删除。

这种多版本时间戳排序机制并不保证可恢复性和无级联性，按照与基本的时间戳排序机制一样的方式对它进行扩展，可以使之成为可恢复的和无级联的。

 

多版本两阶段封锁

多版本两阶段封锁协议 尝试将多版本并发控制的优点和两阶段封锁的优点结合起来。

  • 更新事务执行强两阶段锁，他们持有全部锁直到事务结束
  • 当只读事务开始执行之前，数据库系统通过读取 ts-counter 的当前值来赋予该事务的时间戳。只读事务在执行读操作时准从多版本时间戳排序协议。
  • 当更新事务读取一个数据项时，它获得该数据项上的共享锁并读取该数据项的最新版本。当更新事务想写一个数据项时，它首先要获得该数据项上的排他锁，然后创建一个该数据项的新版本。
  • 当更新事务完成操作后，进入提交阶段，同一时间只允许一个更新事务进行提交。首先 \(T_i\) 将它所创建的每个版本的时间戳设置为 ts-counter 的值加一，然后，\(T_i\) 将 ts-counter 增加 1，并且提交。

 

快照隔离

快照隔离在事务开始执行时给它一份数据库的快照。然后，事务在该快照上已与其他并发事务完全隔离的方式操作。更新数据库的事务与其他更新数据库的事务之间存在潜在的冲突。

更新被控制在事务的私有工作空间中，直到事务通过有效检查，到那时候更新才被写到数据库。

 

多版本

为了实现快照隔离，事务被给予两个时间戳：

  • StartTS(\(T_i\))：为事务\(T_i\) 开始的时间
  • CommitTS(\(T_i\))：为事务\(T_i\) 请求有效性检查的时间

快照隔离是基于多版本的，更新数据项的每个事务都会创建该数据项的一个版本。版本只有一个时间戳，即写时间戳。\(T_i\) 所创建的版本的时间戳被设置为 CommitTS(\(T_i\))

许多实现在事务开始有效性检查之前就创建版本，由于此时版本的时间戳不可用，因此时间戳被设置为无穷大，并在有效性检查是将其更新为正确的值。

当事务 \(T_i\) 读取一个数据项时，具有时间戳 \(\le\) StartTS(\(T_i\)) 的数据项的最新版本将返回给 \(T_i\)。因此，\(T_i\) 不会看到在 \(T_i\) 启动之后才提交的任何事物的更新，但会看到在它启动之前提交的所有事务的更新。

 

有效性检查

两个并发执行的事务可能潜在地更新同一个数据项，由于两个事务以隔离地方式操作并使用它们各自地私有快照。快照隔离有两个变种：先提交者胜 first committer wins 和 先更新者胜 first updater wins

如果事务 \(T_j\) 在从 \(T_i\) 开始执行直到 \(T_i\) 开始有效性检查的任何时刻处于活跃状态或部分提交状态，则称它和事务 \(T_i\) 是 并发 的，形式化的：

  • StartTS(\(T_j\)) \(\le\) StartTS(\(T_i\)) \(\le\) CommitTS(\(T_j\))
  • StartTS(\(T_i\)) \(\le\) StartTS(\(T_j\)) \(\le\) CommitTS(\(T_i\))

  • 对于先提交者胜

    事务 \(T_i\) 开始有效性检查时，以下操作作为分配其 CommitTS 之后作为有效性检查的一部分来执行

    • 检查是否存在与 T 并发的任何事物，对于 T 打算写出的某些数据项，该事务是否已经将更新写入了数据库。可以通过检查 \(T_i\) 打算写出的每个数据项 \(d\) 来完成，检查是否存在数据项 \(d\) 的一个版本的时间戳介于 StartTS 和 CommitTS 之间。
    • 如果发现这样的数据项，则 \(T_i\) 中止
    • 如果没有发现这样的数据项，则 T 提交，并且将其更新写入数据库

  • 对于先更新者胜

    当事物 \(T_i\) 试图更新一个数据项时，它请求该数据项一个写锁：

    如果没有并发事务持有该锁，则：

    • 如果这个数据项已经被任何其他并发事务更新，则 \(T_i\) 中止。
    • 否则 \(T_i\) 能够继续执行其操作，包括可以提交。

    然而，如果另一个并发事务 \(T_j\) 已经持有该数据项的写锁，则 \(T_i\) 不能继续执行，并且：

    • \(T_i\) 等待直到 \(T_j\) 中止或提交
    • 如果 \(T_j\) 中止，则该锁被释放，\(T_i\) 可以获得该锁。此时，如果有另一个并发事务更新过该数据项，则 \(T_i\) 中止；否则 \(T_i\) 执行其操作
    • 如果 \(T_j\) 提交，则 \(T_i\) 必须中止。

 

串行化问题和解决方案

快照隔离不能保证可串行化

考虑如下情形

无论再先提交者胜或者先更新者胜的情况下，其都能够并发进行并提交

但是这样会导致 A 和 B 的值交换，是不符合逻辑的。

一对事务中的每一个都读取对方所写的一个数据项，但是两个事务所写的数据项集合并不存在任何共同的数据项，这种情况称为 写倾斜 write skew

 

可串行化快照隔离 Serializable snapshot ioslation (SSI)

其直观意义如下：假设我们跟踪事务之间的所有冲突（即写-写、读-写、写-读冲突），如果事务 \(T_1\) 和 \(T_2\) 再一个元组上有冲突的操作并且 \(T_1\) 的操作在 \(T_2\) 之前，那么我们可以构造一个事务的优先图，图中有一条从 \(T_1\) 到 \(T_2\) 的有向边。确保可串行化的一种方式是在事务优先图中查找环路，并在找到环时回滚事务。

由于快照隔离导致不可串行化的主要原因是读写冲突。在快照隔离允许不可串行化调度的所有情况下，一定存在一个事务同时具有读写冲突的入边和读写冲突的出边。

因此，可串行化快照隔离实现跟踪并发事务之间的所有读写冲突，以检测是否有一个事务同时具有读写冲突的入边和出边的情况。如果检测到这种情况，则会回滚读写冲突中所涉及的一个事务。
